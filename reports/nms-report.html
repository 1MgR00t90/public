🛡️ Penetration Test Report: Network Management System – Stored/Reflected XSS

🧪 Preface

This report documents a manual Web Application pentest conducted with explicit permission from the system owner. The target of the assessment was a network management system application developed using ClaudeAI. The findings underscore the risks of deploying AI-assisted or auto-generated systems without formal security review, as such tools may introduce exploitable flaws if left unchecked.

⚠️ Disclaimer: This test was performed on a development environment solely for educational and learning purposes. No production data or infrastructure was affected. All identifying details have been anonymized.

🧭 System Profile

The system uses JavaScript on the frontend and Flask (Python) on the backend, integrating with multiple cloud-based network APIs. It aggregates data into dashboards for Network Operations Center (NOC) use.

📍 Summary

Affected Path & Endpoint

  • Path: /firewall
  • API Endpoint: /api/firewall/template/{template}/update

Vulnerability Types

  • Stored (Persistent) XSS
  • Reflected XSS
  • HTML Injection

The comment field in the firewall rule editor accepts unsanitized input, which is both reflected in real-time and persisted in future responses.

🧪 Proof of Concept

1. Reflected XSS

  • Payload: <script>alert(1)</script>
  • Behavior: Reflected in response after form submission

2. Stored XSS (Persistent)

  • Same payload gets saved and displayed to other users

3. HTML Injection for External Links

  • Payload: <a href="https://evil.com">Click me</a>
  • Behavior: Link is rendered and clickable, enabling phishing or redirection

🎯 Payload Execution Sequence

A malicious comment is stored, and later inserted unsafely into the DOM using an unsafe template literal. 

function createRuleRow(rule) {
    return `<td>${rule.comment || ''}</td>`; // 🚨 No sanitization
}

🔥 Impact

  • Credential theft via phishing
  • Session hijacking via XSS
  • CSRF-style forced actions
  • Redirection to attacker-controlled sites

🛠️ Recommendations & Remediation

✅ Input Validation

  • Enforce input validation on server side
  • Disallow special characters unless required
  • Use JSON schema or backend validators

✅ Output Encoding / Escaping

function escapeHtml(text) {
    return text
        .replace(/&/g, "&")
        .replace(//g, ">")
        .replace(/"/g, """)
        .replace(/'/g, "'");
}

function createRuleRow(rule) {
    return `<td>${escapeHtml(rule.comment || '')}</td>`;
}

✅ Content Sanitization

  • Use DOMPurify (client) or Bleach (Python)
  • Whitelist only safe tags (e.g. <b>, <a>)
  • Ensure rel="noopener noreferrer" on <a> tags

✅ Security Headers

Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'none';
X-Content-Type-Options: nosniff
X-Frame-Options: DENY

Also set cookies with HttpOnly, Secure, and SameSite=Strict.

🔚 Conclusion

The application reflects and stores unsanitized user input, allowing both reflected and stored XSS. While the external APIs were secure, the frontend and backend rendering logic failed to sanitize outputs. Despite being in a dev environment, this flaw could lead to serious compromise in production scenarios.

The vulnerability was responsibly reported and fixed via sanitization and validation.
